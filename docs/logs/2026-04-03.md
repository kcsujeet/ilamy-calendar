# Development Log - 2026-04-03

## Changes

- **[resource-calendar]**: Added `data` field to `Resource` interface.
  - This allows users to pass custom metadata (e.g., avatar URLs, roles) per resource.
  - The metadata is accessible in custom resource renderers (`renderResource`).
  - Matches the `data` field pattern used in `CalendarEvent`.
- **[documentation]**: Updated `docs/types-and-interfaces.md` to include the new `data` field in the Resource table.
- **[calendar-engine]**: Added visible date range to `onDateChange` event.
  - Signature updated to `(date: Dayjs, range: { start: Dayjs; end: Dayjs }) => void`.
  - Enables efficient backend fetching of events for the currently visible period.

- **[perf]**: Fixed performance regression from PR #92 — multiple issues:
  1. `normalizeEvents` called on every render, creating new array references → forced `isDeepEqual` to run expensive recursive deep comparison on every render. Fix: memoized with `useMemo`.
  2. Both `CalendarProvider` and `ResourceCalendarProvider` used `calendarEngine` object (new every render) as a single `useMemo` dep → context value always recomputed → all consumers re-rendered on every prop change. Fix: destructured engine return, used individual stable values as deps.
  3. `fixTimezoneOffset` plugin's `restoreTimezone` created a new `dayjs.tz()` instance on every `startOf()`/`endOf()` call. Fix: added offset cache (`Map<string, number>`) so repeated lookups for the same local time + timezone skip the expensive construction. Warm cache brings overhead from ~15% to ~1.5%.
  4. Demo page `businessHours` array recreated on every render, invalidating context memoization. Fix: memoized with `useMemo`.

## Notes

- Used `Record<string, any>` for the `data` field to provide maximum flexibility, with appropriate lint ignore comments for both `oxlint` and `biome`.
- Verified changes with 748 tests passing.
- **Perf fix round 1**: `isDeepEqual` from PR #92 ran expensively due to `normalizeEvents` creating new objects every render. Offset cache in `restoreTimezone` brings re-render overhead from ~15% to ~1.5%.
- **Perf fix round 4 — root cause**: PR #92 replaced `export default dayjs` with a `timezoneAwareDayjs` wrapper that routes ALL `dayjs()` calls through `dayjs.tz()`. Benchmarked: `dayjs.tz()` is ~600x slower than `dayjs()` due to IANA timezone lookups. Fix: (a) skip `.tz()` entirely when no default timezone is set, (b) cache no-arg `dayjs()` ("now") results for 50ms so dozens of calls within a single React render share one expensive `.tz()` call. Test suite itself went from ~16s to ~12s.
- **Perf fix round 2**: Reduced exit animation from 0.15s to 0.08s for snappier view transitions. Memoized `rows` in MonthView, `firstCol`/`columns` deps in WeekView. Fixed bogus `visibleDays.map` useMemo dep. Memoized demo page `classesOverride`, `renderEvent`, `renderCurrentTimeIndicator` with `useCallback`/`useMemo`.
- **Perf fix round 3 — resource week view**: Root cause: Dayjs objects fail React reference equality, so `useProcessedWeekEvents`/`useProcessedDayEvents` useMemo deps were ALWAYS stale, running the expensive event positioning algorithm on every render for every resource (6×). Fixes:
  - Added `useStableDays()` hook to both processed-event hooks — stabilizes Dayjs[] references via `useRef` + timestamp comparison
  - Used `.valueOf()` numeric surrogates for Dayjs deps in `events` useMemo
  - Memoized `columns` and `rows` in `ResourceEventGrid`; stabilized `getVisibleResources()` result via `useRef`
  - Replaced ~200 AnimatedSection (AnimatePresence + motion.div) components in resource view headers with plain divs — eliminated massive Framer Motion overhead for static header cells
- Verified with 756 tests, 0 lint errors.
- **Perf fix round 5 — useMemo Dayjs dep leak + HorizontalGrid key**: Three remaining bottlenecks:
  1. `useProcessedWeekEvents` and `useProcessedDayEvents` computed `weekStart`/`weekEnd` (via `.startOf()`/`.endOf()`) OUTSIDE the `events` useMemo, then included them as deps alongside numeric timestamps. Since `.startOf()`/`.endOf()` always return NEW Dayjs instances, the memo busted on every render — the expensive event positioning ran for every resource row on every render. Fix: moved the `.startOf()`/`.endOf()` calls INSIDE the useMemo and use the stable `days` array as the dep.
  2. `resource-month-horizontal.tsx` still used AnimatedSection (AnimatePresence + motion.div) for ~35 month header cells. Replaced with plain divs (consistent with round 3 fix for other resource headers).
  3. `horizontal-grid.tsx` had `key={currentDate.format('YYYY-MM')}` on the inner div wrapping all rows, forcing React to fully destroy and recreate all children on month navigation instead of reconciling. Resource rows have stable keys (room-a, room-b), so reconciliation is much cheaper. Removed the destructive key.

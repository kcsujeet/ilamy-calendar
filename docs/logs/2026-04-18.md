## Key factory (branch `perf/reduce-bundle-duplication`)

- **[lib/utils]**: Added `keys.ts` factory centralizing all key/id/data-testid generation — inspired by TkDodo's React Query key factory pattern (https://tkdodo.eu/blog/effective-react-query-keys). Before this, ~100 call sites across 15+ files built strings ad-hoc via template literals. Single source of truth prevents format drift and makes cross-cutting renames one-point.
- **Factory categories**:
  - `keys.col.*` — grid column config ids (`time-col`, `day-col-{YYYY-MM-DD}`, `day-col-{YYYY-MM-DD}-resource-{id}`, ...)
  - `keys.cell.*` — per-cell test-ids (`day-cell-{YYYY-MM-DD}`, `vertical-cell-{YYYY-MM-DD}-{HH}-{mm}-{resource?}`, ...)
  - `keys.container.*` — wrapper test-ids (`vertical-col-{id}`, `horizontal-row-{id}`, `horizontal-event-{id}`, ...)
  - `keys.header.*` — header test-ids (`resource-week-time-label-{HH}`, `year-day-{monthKey}-{dayKey}`, ...)
  - `keys.listKey.*` — React `key=` values for mapped items (ISO-based to stay unique across DST boundaries)
  - `keys.droppable.*` — dnd-kit droppable ids (MUST use ISO timestamp per registry uniqueness requirement)
- **Tests**: 21 new tests covering every factory branch, including the allday+resource suffix ordering for droppable ids.
- **Status**: Factory + tests landed. Migration of call sites is next — paused for design review (string output vs array, constants vs functions, single `keys` namespace vs split `testIds`/`keys`).
- **Restructured after design feedback**: reorganized the factory so the PURPOSE of the string is the primary axis and the concept is secondary. Purpose tiers:
  - `keys.id.*` — runtime identifiers (grid config column ids, dnd-kit droppable registry ids)
  - `keys.testId.*` — values for `data-testid` attributes (cells, containers, headers)
  - `keys.reactKey.*` — values for React `key=` props on mapped lists
  - Example: `keys.id.col.day(day)` vs `keys.testId.cell.day(day)` vs `keys.reactKey.weekHeaderDay(day)`. Makes the INTENT of every string clear at the call site.
- Tests restructured to match (still 21 tests, all passing).
- **Further simplification pass** across all three tiers (`id`, `testId`, `reactKey`):
  - Merged `id.col.day` + `dayResource` into one function with optional `resourceId` arg.
  - Merged `testId.cell.day` + `dayHour` into one function with optional `hour`/`minute` args.
  - Grouped `testId.container` by orientation (`vertical.{col,events}`, `horizontal.{row,rowLabel,events,event}`).
  - Grouped `testId.header` by view context (`resource.*`, `week.*`, `month.*`, `year.*`).
  - Grouped `reactKey.weekHeader.{day,hour,resource}`; merged `weekAllDayRow`+`resourceAllDayRow` into a single `allDayRow` (React keys only need scope-uniqueness so two variants collapsed cleanly).
  - Naming is now consistent: concept-first word order, same nesting pattern at every level.
  - 22 tests passing after restructure.
- **Flattened the factory** (final): dropped the `id.` / `testId.` / `reactKey.` top-level wrappers. Each concept now sits at the root and returns a semantic string whose prefix identifies it (e.g. `day-col-`, `day-cell-`, `week-header-day-`). The call site chooses the role — same string can serve as a React `key=`, an `id`, or a `data-testid`. Rationale: the string prefix already encodes the concept; the extra purpose-tier was redundant. Shape: `keys.col.*`, `keys.cell.*`, `keys.container.vertical/horizontal.*`, `keys.header.resource/week/month/year.*`, `keys.weekHeader.*`, `keys.allDayRow`, `keys.resourceCell`, `keys.droppable.dayCell`.

### Biome enforcement rule

Added a GritQL plugin at `biome-plugins/no-inline-template-keys.grit` that flags inline template literals in JSX `id`, `key`, `data-testid`, `htmlFor`, and `transitionKey` attributes. Prevents future drift back to ad-hoc string building.

**Scoping:** Biome 2.4.9 doesn't support disabling plugins in `overrides[].plugins` — a confirmed limitation (biomejs/biome#6851). Workaround: declare the plugin inside an override with negated `includes`. The plugin is active on library source files and skipped for `*.test.*`, `*.spec.*`, and `src/components/demo/**` (test scaffolding and showcase code don't need factory discipline).

**Pattern:**
```
or { `id={$expr}`, `key={$expr}`, `data-testid={$expr}`, `htmlFor={$expr}`, `transitionKey={$expr}` } as $match where {
    $expr <: r"`.*`",
    register_diagnostic(span=$match, message="...")
}
```

Uses regex matching on the captured expression source to detect template-literal values. Static strings (`key="foo"`) and factory calls (`key={keys.listKey(...)}`) pass through.

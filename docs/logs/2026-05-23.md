# Development Log - 2026-05-23

## Changes

- **[review-only session]**: Drafted code review for PR #169 (week-view alignment, follow-up to PR #125). No production code changed today on this branch.
- **[hooks / pr-post approval]**: Added a PreToolUse Bash hook (`.claude/hooks/check-pr-post-approval.sh`) that blocks public-facing GitHub posting commands (`gh api .../pulls/.../{comments,reviews} -X POST`, `gh pr {comment,review,create}`, `gh issue {comment,create}`) unless the same Bash command chains the ritual `touch .claude/state/pr-post-approved.flag &&` ahead of the gh part. Why: a recurring failure mode where stale or implied approval gets interpreted as fresh. The ritual makes approval explicit, deliberate, and auditable in the Bash history. First iteration used a 30-second file-mtime window; live testing immediately showed that doesn't work because PreToolUse hooks fire BEFORE the Bash command runs, so the chained touch hasn't happened yet when the hook checks the file. Switched to substring detection of the touch literal in the command text. Registered in `.agents/settings.json` under `hooks.PreToolUse` with `matcher: "Bash"`. Updated the code-review skill (`.claude/skills/code-review/SKILL.md`) with the mandatory ritual and added a corresponding hard rule to `AGENTS.md`. Hook contract verified against https://code.claude.com/docs/en/hooks (exit code 2 with stderr blocks the call; `tool_input.command` is read from stdin via `jq`). Tested all paths: protected-no-touch block, protected-with-inline-touch allow, read-only gh allow, unrelated bash allow. Plus a live `gh pr create` confirmed the block + ritual flow end-to-end through Claude Code's actual tool plumbing.

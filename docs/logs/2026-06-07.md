# Development Log - 2026-06-07

## Changes

- **[scripts/release]**: Added root release scripts so the library publishes from
  the repo root: `build:lib` (build only `@ilamy/calendar`), `publish:lib`
  (`cd packages/calendar && bun publish`), `release` (build:lib + publish:lib),
  and `release:dry` (build + `bun publish --dry-run`). `bun publish` honors
  `publishConfig.access: public` and runs prepack/postpack. Verified `release:dry`
  from root packs only `@ilamy/calendar` and leaves the source package.json intact.

- **[publish/single-package]**: Reshaped the publish surface to a **single
  published package, `@ilamy/calendar`**, keeping the monorepo for development.
  - `@ilamy/types`, `@ilamy/utils`, `@ilamy/ui`, `@ilamy/calendar-recurrence` are
    now `private: true` (never published); calendar's bunup `noExternal`s them
    (`/^@ilamy\/(ui|utils|types|calendar-recurrence)/`) so they're bundled in.
  - Recurrence ships as the subpath `@ilamy/calendar/plugins/recurrence` (new
    entry `src/plugins/recurrence.ts` re-exporting the bundled plugin). bunup
    keeps `@ilamy/calendar` external so the subpath self-references the host at
    runtime (one shared context instance).
  - Design tokens moved `@ilamy/ui/src/styles/globals.css` → `packages/calendar/
    styles/globals.css`, shipped as `@ilamy/calendar/styles.css`.
  - calendar `dependencies` now include the deps the bundled code pulls in (clsx,
    tailwind-merge, class-variance-authority, the `@radix-ui/*` set, rrule) —
    verified to exactly match the dist's runtime imports. `@ilamy/*` workspace
    packages are devDependencies (bundled from source). `prepack` now also strips
    devDependencies from the published manifest (they reference unpublished
    private packages).
  - Demo repointed: recurrence from `@ilamy/calendar/plugins/recurrence`, styles
    from `@ilamy/calendar/styles.css`, a single `@source .../@ilamy/calendar/dist`.
  - Verified: `bun publish --dry-run` packs only `@ilamy/calendar` (private
    packages error out); packed manifest is clean (no `@ilamy/*`, `sideEffects:
    false`, the four entry points). Full clean `bun run ci` green.

- **[ci/parity]**: Reordered the root `ci` script to `check && build && type-check
  && test` (build now runs first). On a fresh checkout there is no `dist/`, and
  the recurrence plugin + demo resolve `@ilamy/calendar` (and `@ilamy/ui`,
  `@ilamy/calendar-recurrence`) through their built `dist/*.d.ts` — so type-check
  and test must run after build. GitHub CI failed (`Cannot find module
  '@ilamy/calendar'`, plus downstream implicit-any / unused-`@ts-expect-error`)
  while local passed only because stale `dist/` existed. Reproduced by `rm -rf
  packages/*/dist apps/*/dist` then `bun run type-check`; build-first makes it
  green. (Build order doesn't need dist among siblings — bunup externalizes the
  `@ilamy/*` imports — so a clean `bun run build` succeeds.)

  - **apps/demo carve:** moved the demo (`src/components/demo`, `App.tsx`,

  - **Root/CI:** added root `dev` → `@ilamy/demo`; `build`/`type-check` now use

- **[monorepo/phase-3]**: Extracted two shared leaf packages to finish the plugin
  boundary and kill duplication.
  - `@ilamy/utils` — the configured `dayjs` instance (`./dayjs`) plus shared
    helpers `safeDate`/`omitKeys`/`listKey` (`./helpers`). Created because the
    configured dayjs's plugin type augmentations (`.utc()`, `.tz()`,
    `.isSameOrBefore()`) are **lost across the dist `.d.ts` boundary**: bunup
    emits the export as plain `typeof dayjs`, so any consumer of
    `@ilamy/calendar`'s `dayjs` export (incl. third-party plugins) lost those
    methods. Plugins now import dayjs from `@ilamy/utils/dayjs` (source) and get
    the augmentations. Also dedup'd recurrence's local helper copies.
  - `@ilamy/types` — the plugin-contract types (`CalendarEvent`, `WeekDays`,
    `BusinessHours`, `IlamyPlugin`, `PluginMutationArgs`, `PluginDateRange`,
    `PluginView`, `EventFormSlotContext`, `EventMutationScopeSlotContext`,
    `Dayjs`). Canonical definitions live here; `@ilamy/calendar` re-exports them
    from the old `@/` locations (thin shims) so core call sites are unchanged.
- **[monorepo/recurrence]**: `@ilamy/calendar-recurrence` now resolves
  `@ilamy/calendar` via its built dist `.d.ts` (faithful dogfooding — the
  164-error "@/ flood" from pointing at source is gone). Contract types are
  imported from `@ilamy/calendar` (NOT `@ilamy/types`) because the
  `CalendarEvent` `declare module` augmentation must target the package whose
  runtime exposes events; bunup inlines `CalendarEvent` into calendar's dist, so
  the runtime (`useIlamyCalendarContext().rawEvents`) and the augment must agree
  on that identity.
- **[recurrence/ui]**: Rewrote the recurrence editor's frequency `Select` to the
  Radix compound API (`SelectTrigger`/`SelectValue`/`SelectContent`/`SelectItem`)
  and the edit dialog to the Radix `Dialog` API (`onOpenChange`, no `onClose` on
  `DialogContent`).
- **[recurrence/tests]**: Fixed `bun:test` type resolution (added
  `bun-env.d.ts` with `/// <reference types="bun" />`; dropped the package-local
  `@types/bun` so it uses the hoisted root copy). Updated `selectFrequency` to
  drive the Radix Select (open trigger, click option). Converted the 7
  controlled-`rerender` editor tests to unmount+remount: happy-dom + Radix +

- **[calendar/tests]**: Removed the 4 recurrence-integration tests from

  source/dist dual-`CalendarContext` split (calendar source provider vs. the
  plugin's dist-context hook). Plugin integration is covered in the recurrence

- **[build/lint]**: calendar bunup externalizes all `@ilamy/*` siblings.
  biome `noRestrictedImports` (the dayjs ban) now exempts
  `packages/utils/src/dayjs.ts` and `packages/types/src/index.ts`.

# Development Log - 2026-05-31

## Changes

- **[refactor / recurrence-plugin contract redesign — generic hooks]**: Reworked the plugin contract (`src/lib/plugin.ts`) from recurrence-shaped hooks to generic ones, after a design discussion benchmarking Vite/Rollup and FullCalendar plugin systems. Replaced `expandEvent` (per-event) with `transformEvents` (sequential list-transform chain, Rollup `transform` kind), `ownsEvent` with `claimsEvent` (first-match), unified `applyEdit`/`applyDelete` on a `PluginMutationArgs` arg, and collapsed `renderEditDialog` + `renderFormSection` into a single `renderSlot`. The edit-scope dialog folded into the interaction flow as a slot (gathers the opaque `scope`). Runtime: `expandEvents`→`transformEvents` (sequential reduce, then the core applies the range-overlap filter the old per-event fallback did), `getOwner` unchanged, `getFormSectionPlugins`→`renderSlot` (aggregates keyed nodes). dnd gate moved from `owner?.renderEditDialog` to `owner?.applyEdit` (equivalent for recurrence). Behavior-preserving: type-check clean, 897 pass / 0 fail, build OK, biome clean on changed files.

- **[refactor / plugin kernel relocation + fully-generic slot]**: Two follow-ups applied. (1) Made `renderSlot` fully generic: `renderSlot(slotName: string, context: unknown)` instead of a closed `SlotRequest` discriminated union, so the contract no longer names any feature/slot. The built-in slot names (`event-form`, `event-mutation-scope`) and their context shapes (`EventFormSlotContext`, `EventMutationScopeSlotContext`) moved to a separate `slots.ts` catalog the contract does not depend on; plugins narrow `context` by `slotName` and cast, call sites use `satisfies` for safety. (2) Moved the kernel out of `src/lib/plugin.ts` into a top-level shared `src/plugins/` module (`types.ts`, `create-plugin-runtime.ts`, `slots.ts`, `index.ts`) — per Bulletproof React's no-cross-feature-import rule the kernel is shared infra (used by the calendar feature, dnd, form), so it cannot live under `src/features/`. All importers retargeted to `@/plugins`. type-check clean, 897 pass / 0 fail, build OK, biome clean.

- **[refactor / plugins live under src/features/plugins]**: Reorganized so `src/features/plugins/` is its own mini-`src` (Bulletproof React applied recursively). The shared kernel moved to `src/features/plugins/lib/` (`types.ts`, `create-plugin-runtime.ts`, `slots.ts` + their test) and the recurrence plugin moved (via `git mv`, history preserved) to `src/features/plugins/recurrence/`. No barrel/index.ts files — every import targets the concrete module (`@/features/plugins/lib/slots`, `@/features/plugins/recurrence/utils/recurrence-handler`, etc.). Retargeted all importers: the 7 kernel consumers split their `@/plugins` import across `lib/types` / `lib/create-plugin-runtime` / `lib/slots`, and the 11 `@/features/recurrence/*` importers plus the two `src/index.ts` re-export paths moved to `@/features/plugins/recurrence/*`. Public API surface unchanged (only internal source paths moved). type-check clean, 897 pass / 0 fail, build OK, biome clean.

- **[refactor / slot catalog is host-owned, not kernel/plugin]**: Moved the slot catalog out of the plugin kernel. `SLOT_EVENT_FORM` / `SLOT_EVENT_MUTATION_SCOPE` + their context types are the host calendar's mount points (the core event form and dnd flow render them), not generic plugin machinery and not recurrence's. Putting them in `features/plugins/lib/` wrongly implied "shared vocabulary across all plugins" (BR: shared/lib = broadly-reused code). They cannot belong to recurrence either: the core event form calls `renderSlot(SLOT_EVENT_FORM, ...)`, so if the constant lived in the recurrence plugin the core would have to import a plugin, defeating the decoupling. Moved them to `src/components/calendar-slots.ts` (host shared component layer); any plugin imports them from there (shared → feature, allowed). `features/plugins/lib/` is now pure generic machinery: the `IlamyPlugin` contract (`types.ts`) and `createPluginRuntime` only. The kernel runtime test no longer references real slot names (uses arbitrary strings) to prove the runtime is slot-agnostic. type-check clean, 897 pass / 0 fail, build OK, biome clean.

- **[drag-to-create / fix]**: Timezone round-trip bug — dragging Jun 1→29 in America/Halifax created an event ending Jun 30 (form showed 3:00 AM / 2:59 AM, the UTC wall-clock). Root cause (proven empirically against the configured dayjs, not assumed): `data-start`/`data-end` are UTC ISO (`toISOString()`), and `@ilamy/utils/dayjs` aliases the `dayjs()` constructor to `dayjs.tz()` (see `dayjs.ts:84`). Under the calendar's `dayjs.tz.setDefault(timezone)` (set in `use-calendar-engine.ts:196`), `dayjs.tz('…T03:00:00Z', tz)` reads the **literal** `03:00` wall-clock and stamps it as `tz` instead of converting the instant — so a 00:00-Halifax cell round-tripped back to 03:00. An interim fix using `dayjs(iso).tz(timezone)` did NOT help (the constructor mis-parse happens before `.tz()`). Correct fix: parse the instant explicitly with `dayjs.utc(iso)`, then `.tz(timezone)` (or `.local()` when no calendar tz). The normal cell-click path is unaffected because it reuses the cell's original Dayjs and never serializes→reconstructs through a `Z` string. The same fix corrects the `isSameRegion` same-day clamp (was comparing the wrong day) and the form's End Date display (the form derives both date and time from this Dayjs; the `toDate()`→`dayjs(jsDate)` path has no Z-string mis-parse). Verified via a node probe across `TZ=America/Halifax` and `TZ=UTC`. The unit test was falsely green because it never called `setDefault`; it now does, reproducing the bug (RED) before the fix (GREEN).

- **[drag-to-create / fix]**: Two mirror-overlay bugs found in the demo (short "Extra Small 300px" calendar). (1) The selection overlay painted far outside the calendar body: the mirror was the union of the start/last cell `getBoundingClientRect()`s, which extend past the clipped (scrolled) calendar body, and the overlay is portaled to `document.body` so the body's overflow can't clip it. Fix: clip the mirror to the `[data-testid="calendar-body"]` ancestor rect. Extracted pure `unionRect(Rect, Rect)` (moved off the old element-based version) and new `intersectRect(Rect, Rect): Rect | null` into `selection.ts`; the provider's `computeMirror` reads the DOMRects and clips. 7 new pure specs in `selection.test.ts` (incl. the exact overflow case, full-edge clip, and zero-area edge-touch → null). (2) Native text selection highlighted event titles/day-numbers during a drag. Fix: on gesture activation, set `user-select: none` (+ `-webkit-user-select`) on `document.body` and clear any in-progress selection; restored in `reset()` (covers commit/cancel/blur/Escape). 31 tests in the package. Note: clipping couples to the existing `data-testid="calendar-body"` marker (the visible calendar bounds for both regular and resource calendars).
- **[drag-to-create / core fix]**: Feature didn't work in vertical daily-granularity views (resource month vertical, week at daily granularity). Root cause: `vertical-grid-col.tsx` passed `hour={day.hour()}` for ALL cells, so even a `gridType="day"` cell described a 00:00-01:00 one-hour range instead of a full day. That made `isFullDaySpan` false, so the plugin's cross-day clamp rejected every other cell and the selection never extended. The horizontal grid already does `hour={gridType === 'hour' ? day.hour() : undefined}` (which is why month worked). Fix: mirror that conditional in `vertical-grid-col.tsx` so daily cells emit a full-day range. Side benefit: click-create on a daily vertical cell now makes a full-day event (consistent with horizontal month) instead of a midnight 1-hour event. TDD: new `vertical-grid-col.test.tsx` case asserts a `gridType="day"` cell's `data-end - data-start` is 23h (RED before, GREEN after); also set the test helper's default `gridType: 'hour'` so the existing hourly/sub-hour tests still exercise hourly behavior.

# Development Log - 2026-05-30

## Changes

- **[dev server / bun → vite]**: Migrated the interactive demo's dev server from Bun's `Bun.serve` (`bun --hot src/index.tsx`) to Vite. Bun's dev server had caching/HMR problems. Added a root `vite.config.ts` with `root: 'src'` so `src/index.html` stays in place and the Bun production server (`bun run start`) keeps working unchanged. The `dev` script now runs `bunx vite`.
- **[deps]**: Added `vite@^8.0.14`, `@vitejs/plugin-react@^6.0.2`, `@tailwindcss/vite@^4.3.0` as devDependencies (latest majors). Verified Vite 8 has no breaking changes for the config options used (`root`, `resolve.alias`, `server.port`/`strictPort`, `plugins`, `defineConfig`). plugin-react 6's `@rolldown/plugin-babel` / `babel-plugin-react-compiler` peers are optional, so no extra installs. Node 23.11 satisfies Vite 8's `>=22.12` engine requirement.
- **[AGENTS.md / hard rules]**: Added an "ALWAYS use the latest published version when adding a dependency" rule (check `npm view`, don't copy versions from examples/memory). Reconciled the existing "NEVER use npm/node/vite/pnpm" rule, which now contradicted the Vite-based dev server — reworded to "npm/node/pnpm as the package manager or runtime" and noted tools run via `bunx` (e.g. `bunx vite`).
- **[examples / latest versions]**: Bumped all three examples (`vite`, `nextjs`, `astro`) to latest deps. Notable majors: Vite 6→8 + `@vitejs/plugin-react` 4→6 (vite example), Astro 5→6 + `@astrojs/react` 4→5 (astro example), TypeScript 5.9→6, `@types/node` 22→25 (nextjs), `@ilamy/calendar` → ^1.7.0 everywhere. Verified peers (plugin-react 6 ↔ vite 8, @astrojs/react 5 ↔ react 19, next 16 ↔ react 19; Astro 6 engine node ≥22.12, local 23.11). Refreshed each example's `bun.lock` and confirmed all three `bun run build` successfully.
- **[examples/vite / TS6 fix]**: Added `examples/vite/src/vite-env.d.ts` (`/// <reference types="vite/client" />`). TypeScript 6's stricter side-effect-import checking (TS2882) failed the build on `import './index.css'` because nothing referenced Vite's client type shims (which declare `*.css`). This is the standard Vite scaffolding file that was missing.
- **[resource week view / date format] (issue #157)**: Resource Week View day headers showed the `M/D` slash format (e.g. `5/19`) while the standard Week View shows only the day number (`19`). Made them consistent by switching the resource week headers to `day.format('D')`. Fixed all three render paths: horizontal day header, vertical hourly day header, and the vertical daily-mode first column.
- **[now-line / day-resolution vertical views] (issue #123)**: The current-time indicator rendered a meaningless sub-day stripe in day-resolution vertical views (Resource Month Vertical, Resource Week Vertical daily). `VerticalGridEventsLayer` mounted `CurrentTimeIndicator` without a `gridType` gate. PR #122 had added the `showNowLine = gridType === 'hour' && ...` gate to the *horizontal* events layer but left the vertical one ungated. Mirrored the same gate into `vertical-grid-events-layer.tsx`.
- **[feat / isCellDisabled] (issue #79)**: Added an `isCellDisabled?: (info: CellInfo) => boolean` prop to `IlamyCalendar` and `IlamyResourceCalendar`. Returning `true` disables a cell — blocks event-creation clicks, rejects drag-drops, and grays it out, reusing the single existing `disabled` lever in `DroppableCell`. Composes with `businessHours` via OR. `DroppableCell` is now the sole authority for the final disabled state: it computes the slot `start`/`end` once, resolves the full `resource` via `getResourceById` (undefined on regular calendars), builds the `CellInfo`, and ORs `disabled || isCellDisabled?.(info)`. **BREAKING:** renamed the public `CellClickInfo` type to `CellInfo` (the `CellClickInfo` name is removed entirely — no alias) and **removed `resourceId` from the cell-info payload in favor of the full `resource` object** — `onCellClick` and `isCellDisabled` now expose `info.resource` (use `info.resource?.id` instead of `info.resourceId`). Warrants a major version bump. Scope decision: shipped the disabling predicate only; `renderCell` (custom content) deferred — it can't block drops and wasn't required by #79. TDD: failing tests first in `droppable-cell.test.tsx`, then integration tests through both calendars. 886 tests pass. Code-reviewed via the review skill: one finding applied — the regular-calendar `onCellClick` tests asserted the now-removed `callArgs.resourceId`, updated to `callArgs.resource` so they validate the new shape instead of passing vacuously. Then refactored `droppable-cell.tsx` for clarity: extracted a pure `getCellRange(date, hour, minute)` helper (immutable, early returns) replacing the mutable `let end` cascade, removed the now-unneeded file-level `oxlint-disable no-negated-condition`, and named the className conditions (`clickBlocked`, `showDropHighlight`, `disabledClassName`). Behavior unchanged; 886 tests pass. Further cleanup after review feedback: widened the internal `getResourceById` to accept an optional id (with an early return when undefined) so the resource resolution in `DroppableCell` collapses to one line (`getResourceById?.(resourceId)`) instead of a thin `resolveResource` wrapper or a multi-line ternary; named the disabled-cell class (`disabledClass`) to keep the `cn()` args single-line. Also added a code-review rule: ternaries must be single-level AND single-line.
- **[feat / recurrence-plugin phase 1, task 1]**: Added the core plugin contract and runtime resolver in `src/lib/plugin.ts`. Defines the `IlamyPlugin` interface (optional `expandEvent`, `ownsEvent`, `applyEdit`, `applyDelete`, `renderEditDialog`, `renderFormSection`) plus supporting types (`PluginDateRange`, `PluginEditArgs`, `PluginDeleteArgs`, `PluginEditDialogProps`, `PluginFormSectionProps`). The core stays agnostic of any specific plugin; `scope` is opaque to the core. `createPluginRuntime(plugins)` returns a `PluginRuntime` with `expandEvents` (first plugin whose `expandEvent` returns occurrences wins; otherwise falls back to the `eventOverlapsRange` default), `getOwner` (first plugin whose `ownsEvent` claims the event), and `getFormSectionPlugins`. TDD: wrote `src/lib/plugin.test.ts` first (5 tests), confirmed it failed on the missing module, implemented, 5 pass. `eventOverlapsRange(event, start, end)` signature in `src/lib/utils/event-utils.ts` matched expectations. type-check clean.
- **[feat / recurrence-plugin phase 1, task 2]**: Added `src/features/recurrence/recurrence-plugin.tsx` — a `recurrencePlugin()` factory returning an `IlamyPlugin` that adapts the existing recurrence handler functions (`generateRecurringEvents`, `updateRecurringEvent`, `deleteRecurringEvent`) and editor components (`RecurrenceEditDialog`, `RecurrenceEditor`) to the plugin contract. `ownsEvent`/`expandEvent` key off `event.rrule` presence (deviation from the plan's `isRecurringEvent` delegation, which also keys off `uid`/`recurrenceId` and would have failed the spec'd test). TDD: `recurrence-plugin.test.ts` first (3 tests), confirmed failing on missing module, implemented, 3 pass. type-check clean.
- **[feat / recurrence-plugin phase 1, task 3]**: Routed `use-calendar-engine.ts` through the plugin runtime with no behavior change. Removed the direct `recurrence-handler` value imports; the engine now builds a `pluginRuntime` from a new `plugins?: IlamyPlugin[]` config prop (defaulting to `[recurrencePlugin()]`). `getEventsForDateRange` delegates to `pluginRuntime.expandEvents`, and `updateRecurringEvent`/`deleteRecurringEvent` resolve the owning plugin via `getOwner` and call its `applyEdit`/`applyDelete` (guarded, no-op if unowned). Exposed `getOwner` and `getFormSectionPlugins` on the engine return. Parity verified: 520 tests pass / 0 fail across hooks, recurrence, calendar contexts/components, and resource-calendar suites; type-check clean.
- **[release]**: Cut v1.8.0 — `hideExportButton` prop, resources-prop sync (#153), recurrence "edit all" fix (#150), localization across headers/date-picker/recurrence editor (#124), week-column alignment (#160), dynamic body height, event-form focus (#166), today highlight + day-number-only resource week headers (#159/#157), now-line gate in day-resolution vertical views (#123).

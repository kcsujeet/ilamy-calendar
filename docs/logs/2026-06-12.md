  extends `Omit<>`s of the four slice interfaces (omitting slice-internal members and the

  shadowing default.

  `VerticalPositionedEvent` (`kind: 'vertical'`, required `top`/`height`, optional

  (computeColumnSpan already clamps), and the 6-positional-arg `place()` takes an options

  derive `resource.id`); `HorizontalRowSpec.id` unified to `string`.

  (filters read the events' own resource fields; only `getResourceById` touches

## Changes (issue #162 — recurring edit handlers, re-impl of PR #191 on v2)

- **[recurrence / callbacks]**: A scoped recurring edit now fires persistence
  callbacks with the REAL stored rows, not a synthetic merged UI instance, and no
  longer double-fires. Two layers fixed:
  1. **Wrong payload** — `applyScopedEdit` (`use-calendar-data.ts`) always called
     `onEventUpdate({ ...event, ...updates })` with the merged generated instance.
     It now inspects the plugin result: if a `PluginMutationResult` comes back, it
     dispatches `result.updated → onEventUpdate`, `result.added → onEventAdd`, and
     `setCurrentEvents(result.events)`. A bare `CalendarEvent[]` keeps the old path.
  2. **StrictMode double-fire** — `useScopedEventMutation.handleConfirm` ran the
     apply inside a `setState` updater; StrictMode double-invokes updaters in dev, so
     callbacks fired twice. The apply now runs OUTSIDE the updater, reading the latest
     dialog state via a ref; `setDialogState(CLOSED)` follows.
- **[recurrence-handler]**: `updateRecurringEvent` now returns `PluginMutationResult`
  ({ events, updated, added }) instead of `CalendarEvent[]`. Per-scope: "this" →
  updated:[base+EXDATE], added:[override]; "following" → updated:[terminated],
  added:[newSeries]; "all" → updated:[base], added:[]. Base rows carry the series uid
  so `onEventUpdate` sees a correct uid even when the stored base omitted one.
- **[review-fix, stored-override branch]**: editing an already-detached override
  ("this" scope) now reports BOTH rows — `updated:[updatedBaseEvent, detachedOverride]`,
  `added:[]` — so the base's new EXDATE/uid is persisted (previously only the override
  was reported).
- **[types]**: ONE result type — `PluginMutationResult` defined once in `@ilamy/types`,
  re-exported from `@ilamy/calendar` (so the recurrence package imports it) and widened
  `IlamyPlugin.applyEdit` to `CalendarEvent[] | PluginMutationResult` (additive).
- **[tests]**: both recurrence test files migrated to a local
  `updateRecurringEvent = (...) => updateRecurringEventImpl(...).events` wrapper for the
  array-shape assertions, using `.updated`/`.added` directly for the callback assertions.

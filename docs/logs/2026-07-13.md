# Development Log - 2026-07-13

## Changes

- **[release]**: Merged PR #238 (squash `76703c9`) — the motion-dependency removal from yesterday's session is on main.

- **[business-hours]**: Sub-hour business-hour boundaries (issue #237). `BusinessHours.startTime`/`endTime` now accept `'HH:mm'` strings (FullCalendar-style) alongside numbers; fractional numbers are deliberately NOT supported as decimal hours (`9.25` reading as 9:15 was judged too confusing) — they round to the nearest whole hour, and sub-hour precision is strings-only. All boundary math normalizes to minutes via a single `toBoundaryMinutes` helper (malformed and out-of-range strings — `'9:99'`, `'25:00'` — fall back to the 9/17 defaults; `'24:00'` is a valid end-of-day). New semantics for cell enablement: whole-slot containment — a grid slot is business only if it fits entirely inside a config's range, so slots partially crossing a boundary (e.g. an hour cell over a 9:15 start) are disabled/shaded. Provably identical to the old point check for integer configs on hour grids, so no behavior change for existing users. `getViewHours` now floors/ceils the visible range so the hour rows hosting a sub-hour boundary stay visible under `hideNonBusinessHours`. `GridCell` gained a `slotDurationMinutes` prop (default 60) fed by `VerticalGridCol`, and now passes `minute` + `durationMinutes` into `isBusinessHour` (it previously dropped the minute entirely — sub-hour slots were all evaluated at the top of their hour). TDD: 7 failing tests written first (unit: `business-hours.test.ts`, `view-hours.test.ts`; integration: `business-hours.test.tsx` asserting `data-disabled` per cell and gutter visibility), then implementation. Follow-up: a 20-case table-driven slot-duration × boundary matrix (15/30/60/16-min durations × :15/:30/whole-hour/odd :25-:50-:55 boundaries, degenerate/inverted/zero-width/24:00 ranges, split shifts, adjacent-range seam) with hand-computed expectations, plus a 30-min-slot integration test. The adjacent-range case pins the documented limitation that containment is per config — touching ranges do not merge, so a slot straddling the seam is non-business.

- **[code-review skill]**: Encoded the maintainer's ternary-remedy preferences in `readability.md` (section F): preferred remedy for a multi-line ternary with simple branches is now named consts per branch + one-line ternary (eager evaluation of trivial expressions is an accepted tradeoff, protected from being flagged as an efficiency finding); `if`/`else` (in components: inside `useMemo`) for branches with real logic; conditionally-spread partial options objects are explicitly rejected as a remedy. Verified with a reviewer-agent run against flag/pass fixtures.

## Notes

Containment semantics deliberately round *inward*: on grids coarser than the boundary the partial slot is non-business, so the calendar never permits a booking that violates the real business rule (the failure mode issue #237's reporter had to guard against externally). The escape hatch for exact boundaries is `slotDuration={15}`. Month/year views are unaffected (day-granularity check only). `droppable-cell.tsx`'s `getCellRange` still hardcodes 15-minute slot ends when `minute` is defined — pre-existing quirk, untouched here, worth a follow-up if slot durations other than 15 need exact `CellInfo` ranges.

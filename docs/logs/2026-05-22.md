# Development Log - 2026-05-22

## Changes

- **[i18n / date display]**: Removed custom locale format helpers from `date-utils` (`isDayBeforeMonth`, `formatWeekdayDayMonth`, `formatShortMonthDay`, `formatWeekPickerRange`). Header week/day pickers now use dayjs `format('ll')` so ordering and month names follow the active locale without duplicate `Intl` logic.
- **[header / title-content]**: Week title, day picker rows, and week-range list labels use `day.format('ll')` / `start.format('ll') - end.format('ll')`; dropped `currentLocale` prop threading where no longer needed.
- **[tests]**: Removed obsolete `date-utils.test.ts` suites for the deleted helpers; dropped unused `dayjs/locale/fr.js` test import.
- **[cleanup]**: Removed unused `useMemo` imports in `event-form.tsx` and `year-view.tsx`; unused `WeekDays` import in `ilamy-calendar.tsx`.
- **[header / week picker label]**: After PR #124 the week picker title used `${start.format('ll')} - ${end.format('ll')}` which produced "Apr 26, 2026 - May 2, 2026" (~250px). That length forced the whole header to wrap to two lines in week view at common desktop widths while day / month / year views stayed on one line. Year is already visible in the adjacent year-picker button, so dropping it from the range is not a real information loss. Switched to `Intl.DateTimeFormat.prototype.formatRange` with `{month: 'short', day: 'numeric'}`: returns "Apr 26 – May 2" in English (~120px, U+2013 en-dash) and "26 avr. – 2 mai" in French (locale-correct day-first order, preserving PR #124's i18n intent that `MMM D` violated). Same-month ranges intelligently collapse: "Apr 26 – 28" / "26–28 avr.". MDN: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl/DateTimeFormat/formatRange (baseline since August 2021). Existing `flex-wrap`-based layout in `base-header.tsx` is unchanged: once the title is short, the natural wrap behaves correctly without breakpoint surgery.
- **[hooks / useDateTimeFormatters]**: Wrapped `Intl.DateTimeFormat` behind a calendar-scoped hook in `src/hooks/use-date-time-formatters.ts`. The hook reads `currentLocale` and `timezone` from `CalendarContext` and returns `formatDateRange(start, end)` that is automatically locale- and timezone-aware so callers don't plumb either through props. Designed to grow (single-date format, time format, weekday format) as the codebase migrates off ad-hoc dayjs `.format()` calls for display strings. Kept minimal: no `useMemo`, no options-override param, no interface, no top-level constants. The Intl constructor is cheap, the function isn't stable-identity-sensitive anywhere it's used, and the body is a single expression with no shared helpers worth extracting. `timezone` was already being added to `contextValue` by the provider but missing from the context type; added it to `CalendarContextType` so consumers can read it without a cast. `title-content.tsx` is the first consumer; the prior `crossesMonth ? ... : ...` branching in the week-list popover is gone entirely because `formatRange` collapses same-month ranges itself ("Apr 26 – 28" / "26–28 avr.").

## Notes

`dayjs` locale must be set on the calendar engine (`currentLocale` / `dayjs.locale()`) for `format('ll')` to localize header dates. Prefer one formatting path (dayjs) over parallel `Intl` + custom order helpers. `date-utils.test.ts`: 58 tests pass after helper removal.

## Notes

Phase 4 re-validation: Phase 2 had shipped all three prerequisites (`openEventForm`
carries `resourceId ?? resource?.id`; `allDay: eventData?.allDay ?? false`;
`getEventsForResource` optional publicly). Phase 3 shipped the columns-composition
contract (the plan's risk-note-1 variant) — `supportsResources` documented as "columns()
composes the resource axis", `ViewRenderer` already carried the engine rule + `variant`,
and `ViewConfig` carried `resources`/`orientation` — so the resource arrangements became
column-spec/header producers rather than wrapped components.

Owner decisions implemented: (a) `IlamyResourceCalendar` is a deprecated alias (variant
A), pinned by its full moved suite; (b) forced resource-incapable views (e.g.
`initialView="year"` with resources) render as regular, resource-less views — already
the post-Phase-3 behavior (the fork map had no `year` entry, so the dispatcher's escape
hatch rendered it), now pinned by an explicit test.

Drift recorded: `resource-cell.tsx` stays in shared `components/` (the shared
`HorizontalGridRow` consumes it — the plan's "used only by the arrangements" was
inaccurate); the resource week data hooks dissolved into `week.tsx`'s pure column
builders instead of moving; 8 shared-component test files (grids, all-day-row,
droppable-cell, event-form business hours, resource-cell, current-time-indicator) also
imported `ResourceCalendarProvider` beyond the plan's 8-suite inventory — mechanical
provider swaps, assertions untouched. Root testid of a resource calendar is now
`ilamy-calendar` (migration entry added; nothing asserted the old id).

Verification: `bun run ci` exits 0 — 154 (recurrence) + 782 (calendar) tests, 0 fail
(774 baseline + the resource-axis provider pin, the year pin, 2 dev-warn pins, 4
pipeline pins). Exit greps clean: zero `features/resource-calendar` references, zero
"Phase 4 deletes/flips" comments, zero "lane" vocabulary.

- **[code-review skill / hard rules]**: Added three new hard rules to `.claude/skills/code-review/SKILL.md`: (1) **Layout/sizing changes get a viewport simulation** — compute per-column / per-cell width at 375/600/768/1024px for any PR touching CSS sizes, flex/grid, or positioning, and surface the numbers in the comment. Removing a width floor on a 7-column calendar without responsive view switching is almost always wrong. (2) **Author intent ≠ absence of bug** — author claim that a change was deliberate closes the intent question but not the correctness question; layout / observable-behavior comments stay open on a re-review until verified visually OR until the numbers / walkthrough explain why the change is safe. (3) **On re-review, fetch the comment thread first** — `gh api repos/<o>/<r>/pulls/<N>/comments` before triaging prior findings; author replies and project-owner replies are authoritative for what counts as resolved.
- **[code-review skill / verification checklist]**: Added matching checklist items so the new principles can't be skipped silently: a "re-review thread fetched" item under "Scope and intent", plus "Layout / sizing changes simulated at narrow viewports" and "Author intent treated as not equal to absence of bug" under "Comment quality". Each requires concrete evidence (numbers, walkthrough, or visual verification) before the box can be honestly checked.

- **[drag-to-create / fix]**: Timezone round-trip bug — dragging Jun 1→29 in America/Halifax created an event ending Jun 30 (form showed 3:00 AM / 2:59 AM, the UTC wall-clock). Root cause (proven empirically against the configured dayjs, not assumed): `data-start`/`data-end` are UTC ISO (`toISOString()`), and `@ilamy/utils/dayjs` aliases the `dayjs()` constructor to `dayjs.tz()` (see `dayjs.ts:84`). Under the calendar's `dayjs.tz.setDefault(timezone)` (set in `use-calendar-engine.ts:196`), `dayjs.tz('…T03:00:00Z', tz)` reads the **literal** `03:00` wall-clock and stamps it as `tz` instead of converting the instant — so a 00:00-Halifax cell round-tripped back to 03:00. An interim fix using `dayjs(iso).tz(timezone)` did NOT help (the constructor mis-parse happens before `.tz()`). Correct fix: parse the instant explicitly with `dayjs.utc(iso)`, then `.tz(timezone)` (or `.local()` when no calendar tz). The normal cell-click path is unaffected because it reuses the cell's original Dayjs and never serializes→reconstructs through a `Z` string. The same fix corrects the `isSameRegion` same-day clamp (was comparing the wrong day) and the form's End Date display (the form derives both date and time from this Dayjs; the `toDate()`→`dayjs(jsDate)` path has no Z-string mis-parse). Verified via a node probe across `TZ=America/Halifax` and `TZ=UTC`. The unit test was falsely green because it never called `setDefault`; it now does, reproducing the bug (RED) before the fix (GREEN).

- **[drag-to-create / feature]**: Edge auto-scroll during drag.

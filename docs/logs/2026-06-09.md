# Development Log - 2026-06-09

## Changes

- **[recurrence / callbacks]**: Fixed duplicate firing of `onEventUpdate`/`onEventAdd`/`onEventDelete` for scoped recurring edits in dev.

  **Root cause (two layers):**
  1. **StrictMode double-invocation** — `useScopedEventMutation.handleConfirm` ran `applyScopedEdit`/`applyScopedDelete` inside a `setState` updater. React StrictMode (enabled in the demo via `frontend.tsx`) intentionally runs updaters twice in dev, so every scoped confirm fired persistence callbacks twice.
  2. **Wrong callback payload** — `applyScopedEdit` always called `onEventUpdate` with a synthetic merged UI instance (`{ ...event, ...updates }`, e.g. `recurring-1_2`) before applying the plugin mutation, instead of emitting the real stored rows.

  **Fix:** Move side effects out of the setState updater (ref + apply outside). Route recurring edits through `RecurringUpdateResult` / `PluginMutationResult` so the engine dispatches `updated` → `onEventUpdate` and `added` → `onEventAdd` with persisted event ids.

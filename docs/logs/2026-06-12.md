## Notes

Phase 4 re-validation: Phase 2 had shipped all three prerequisites (`openEventForm`
carries `resourceId ?? resource?.id`; `allDay: eventData?.allDay ?? false`;
`getEventsForResource` optional publicly). Phase 3 shipped the columns-composition
contract (the plan's risk-note-1 variant) — `supportsResources` documented as "columns()
composes the resource axis", `ViewRenderer` already carried the engine rule + `variant`,
and `ViewConfig` carried `resources`/`orientation` — so the resource arrangements became
column-spec/header producers rather than wrapped components.

Owner decisions implemented: (a) `IlamyResourceCalendar` is a deprecated alias (variant
A), pinned by its full moved suite; (b) forced resource-incapable views (e.g.
`initialView="year"` with resources) render as regular, resource-less views — already
the post-Phase-3 behavior (the fork map had no `year` entry, so the dispatcher's escape
hatch rendered it), now pinned by an explicit test.

`HorizontalGridRow` consumes it — the plan's "used only by the arrangements" was
inaccurate); the resource week data hooks dissolved into `week.tsx`'s pure column
builders instead of moving; 8 shared-component test files (grids, all-day-row,
droppable-cell, event-form business hours, resource-cell, current-time-indicator) also
imported `ResourceCalendarProvider` beyond the plan's 8-suite inventory — mechanical
provider swaps, assertions untouched. Root testid of a resource calendar is now
`ilamy-calendar` (migration entry added; nothing asserted the old id).

## Clean-code batch A (feat/v2-overhaul)

- **[render stability]**: The engine composer's return and all four slice returns are now
  `useMemo`d, so the provider's `useMemo([engine, …])` actually holds the context value
  referentially stable across parent re-renders with identical props. `EMPTY_PLUGINS`
  module constant replaces the per-render `plugins = []` destructure default (mirrors
  `EMPTY_RESOURCES`); the provider hoists `EMPTY_EVENTS` for the same reason. Failing-first
  regression test in `provider.test.tsx` (red: two distinct context objects per re-render).
- **[type dedup]**: `CalendarEngineReturn` no longer re-declares ~30 slice signatures — it
  extends `Omit<>`s of the four slice interfaces (omitting slice-internal members and the
  renamed `getAllViews` → `getViews`) plus the plugin-runtime passthroughs. The composer's
  45-line hand mapping became slice spreads + overrides. `setCurrentDate` is now typed as
  the slice's `Dispatch<SetStateAction<Dayjs>>` (truthful widening; runtime unchanged).
- **[provider]**: `useCalendarContextValue` unexported (single consumer in-file);
  duplicate `businessHours`/`dayMaxEvents` context entries dropped (the engine spread
  provides them); `dayMaxEvents` is now defaulted ONCE in the config slice
  (`DAY_MAX_EVENTS_DEFAULT`) — provider prop became optional, `IlamyCalendar` lost its
  shadowing default.
- **[layout]**: `PositionedEvent` is an honest discriminated union —
  `VerticalPositionedEvent` (`kind: 'vertical'`, required `top`/`height`, optional
  `zIndex`) | `HorizontalPositionedEvent` (`kind: 'horizontal'`, required `row` +
  truncation flags). Both layout engines return their variant; the events layers lost
  their `?`-guards; layout tests assert the discriminant. `horizontal.ts` internals:
  `OccupancyGrid` is `boolean[][]`, `GridBounds` renamed to unit-flavored
  (`firstUnit`/`lastUnit`/`unitCount`), the redundant single-unit double clamp removed
  (computeColumnSpan already clamps), and the 6-positional-arg `place()` takes an options
  object.
- **[contract]**: `PluginView.component` is now OPTIONAL — renderer dispatch is spec path
  (`columns` + `layout`) → `component` → `null` with a dev-mode console.warn naming the
  view. Built-in day/week/month specs dropped their self-referential components (year
  keeps its escape-hatch component). Failing-first: TS2741 on a spec view without
  `component`, plus a runtime test rendering one. `PluginView.range`'s config param is
  now `Pick<ViewConfig, 'firstDayOfWeek'>` (what the only call site ever passed).
  `VerticalColumnSpec.resourceId` removed (`resource` is the single carrier; consumers
  derive `resource.id`); `HorizontalRowSpec.id` unified to `string`.
- **[small fixes]**: data-slice resource-utilities comment now states the real mechanism
  (filters read the events' own resource fields; only `getResourceById` touches
  `resources`); `setView` TSDoc names its side effects (onViewChange + onDateChange);
  `openEventForm`'s `as CalendarEvent` cast narrowed into a documented
  `buildEventDraft(Omit<CalendarEvent, 'id'>)` builder; `useIlamyCalendarContext` reuses
  a module `EMPTY_RESOURCES` instead of `resources || []` and dropped the dead
  `?? ''` on `currentLocale`; view-renderer's "PR #190 lesson" and "locked engine rule"
  comments replaced with their load-bearing content.
